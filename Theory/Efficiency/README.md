# Временная сложность алгоритмов

**Временная сложность** — оценка количества операций в зависимости от размера входных данных \( n \).  
Позволяет **до реализации** понять, уложится ли алгоритм в ограничения по времени.

Используется **\( O \)-нотация** (асимптотическая верхняя граница).  
Формально: \( O(f(n)) \) означает, что существуют константы \( c \) и \( n_0 \) такие, что при \( n \ge n_0 \) алгоритм выполняет не более \( c \cdot f(n) \) операций.

---

## Основные оценки (от самых быстрых к самым медленным)

| Сложность | Название | Типичный признак / пример |
|-----------|----------|---------------------------|
| \( O(1) \) | Константная | Явная формула, время не зависит от \( n \). |
| \( O(\log n) \) | Логарифмическая | На каждом шаге данные делятся пополам (основание логарифма не важно). |
| \( O(\sqrt{n}) \) | Корневая | Медленнее логарифма, быстрее линейной. \( n = \sqrt{n} \cdot \sqrt{n} \), можно разбить на группы. |
| \( O(n) \) | Линейная | Один проход по данным. **Часто это наилучшее возможное**, т.к. обычно нужно прочитать каждый элемент хотя бы раз. |
| \( O(n \log n) \) | Линейно-логарифмическая | Эффективная сортировка, операции со сбалансированными деревьями. |
| \( O(n^2) \) | Квадратичная | Два вложенных цикла, перебор всех пар. |
| \( O(n^3) \) | Кубическая | Три вложенных цикла, перебор троек. |
| \( O(2^n) \) | Экспоненциальная | Перебор всех подмножеств. |
| \( O(n!) \) | Факториальная | Перебор всех перестановок. |

---

## Полиномиальные алгоритмы и NP-трудные задачи

Алгоритм **полиномиальный**, если его сложность \( O(n^k) \) для некоторой константы \( k \).  
Все приведённые выше оценки, кроме \( O(2^n) \) и \( O(n!) \), являются полиномиальными. На практике \( k \) обычно невелико, поэтому такие алгоритмы справляются с большими данными.

> **Важно:** Существуют **NP-трудные** задачи, для которых полиномиальный алгоритм **неизвестен**. Для больших \( n \) их эффективное решение маловероятно.

## Как оценить, уложится ли алгоритм по времени?

Современный компьютер выполняет ≈ **несколько сотен миллионов** простых операций в секунду.

### Пример:
- \( n = 10^5 \)
- \( O(n^2) \) → \( 10^{10} \) операций → десятки секунд (слишком медленно).
- \( O(n \log n) \) → ≈ \( 1.6 \cdot 10^6 \) операций → укладывается в 1 секунду.

### Таблица‑подсказка (при ограничении 1 секунда)

| Размер \( n \) | Ожидаемая сложность |
|----------------|----------------------|
| \( \le 10 \)   | \( O(n!) \)           |
| \( \le 20 \)   | \( O(2^n) \)          |
| \( \le 500 \)  | \( O(n^3) \)          |
| \( \le 5000 \) | \( O(n^2) \)          |
| \( \le 10^6 \) | \( O(n \log n) \) или \( O(n) \) |
| Очень большой | \( O(1) \) или \( O(\log n) \) |

> **Важно:** Это **приблизительные** ориентиры, реальное время зависит от скрытых констант. Алгоритм с \( O(n) \) может выполнять \( n/2 \) или \( 5n \) операций — разница ощутима.

## Тонкости \( O \)-нотации

- \( O(f) \) даёт **верхнюю** границу. Можно сказать, что линейный алгоритм имеет \( O(n^2) \), но это **вводит в заблуждение** — всегда давайте максимально точную оценку.
- **Нижняя граница** — \( \Omega(f) \) (алгоритм выполняет не менее \( c \cdot f(n) \) операций).
- **Точная граница** — \( \Theta(f) \) (одновременно \( O(f) \) и \( \Omega(f) \)).

## Оптимизация кода (детали реализации)

Даже при одинаковой асимптотике реальная скорость может сильно различаться.

### Поиск элемента (break)
```cpp
// Без break — всегда O(n)
for (int i = 0; i < n; i++) if (a[i] == x) ok = true;

// С break — часто быстрее, но асимптотика та же O(n)
for (int i = 0; i < n; i++) {
    if (a[i] == x) { ok = true; break; }
}
```
> **Сторожевое значение** (добавить `x` в конец) — изящно, но на практике не даёт выигрыша, так как основное время уходит на доступ к памяти, а не на проверку границ.

### Роль компилятора
Современные компиляторы (например, `g++ -O2`) автоматически:
- заменяют деление на сдвиг,
- используют битовые операции,
- оптимизируют вычисления адресов.

Посмотреть ассемблерный код можно через флаг `-S` или на [Compiler Explorer](https://godbolt.org/).

### Влияние кеша процессора
Доступ к памяти **последовательно** (по возрастанию адресов) значительно быстрее, чем **в случайном порядке**, из‑за кеширования.

**Пример:** обход двумерного массива по строкам — быстро, по столбцам — медленно.
```cpp
// Дружелюбно к кешу (быстро)
for (int i = 0; i < n; i++)
    for (int j = 0; j < n; j++)
        s += x[i][j];

// Враждебно к кешу (медленно)
for (int i = 0; i < n; i++)
    for (int j = 0; j < n; j++)
        s += x[j][i];
```
Массив хранится в памяти строками, поэтому первый вариант эффективнее.

### Распараллеливание на уровне инструкций
Современные процессоры могут выполнять **независимые** команды параллельно. Это позволяет ускорить код, разбив вычисления на несколько независимых переменных.

**Пример:** вычисление факториала по модулю \( M \) с использованием двух независимых произведений.

```cpp
ll f1 = 1, f2 = 1;
for (int i = 1; i <= n; i += 2) {
    f1 = (f1 * i) % M;          // нечётные множители
    if (i + 1 <= n)
        f2 = (f2 * (i + 1)) % M; // чётные множители
}
ll f = (f1 * f2) % M;
```
- В теле цикла изменяются две **независимые** переменные \( f_1 \) и \( f_2 \).
- Процессор может выполнять обе операции умножения одновременно (или почти одновременно), что даёт ускорение до **2×**.
- Можно использовать 4 или 8 переменных для ещё большего ускорения (но с учётом ограничений по количеству регистров).
